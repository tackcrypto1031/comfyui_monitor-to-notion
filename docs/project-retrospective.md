## 專案概述

### 目標

建立一個本地桌面應用，用於監控內網中多台運行秋葉整合包的 ComfyUI 電腦的運行狀態，並自動同步記錄到 Notion 資料庫。

### 核心價值

即時、準確地監控內網多台 ComfyUI 電腦的狀態變化，並自動記錄到 Notion 形成可追溯的運行歷史。

### 目標用戶

- 使用多台 ComfyUI 節點的 AI 創作者
- 需要追蹤 ComfyUI 運行歷史的團隊
- 使用 Notion 管理工作流程的用戶

## 技術架構

### 技術選型

| 類別 | 技術 | 原因 |
|------|------|------|
| **桌面框架** | Electron 28.x | 跨平台、開發快速、生態成熟 |
| **前端框架** | React 18.x | 組件化、生態豐富、易於維護 |
| **語言** | TypeScript 5.x | 類型安全、開發體驗好 |
| **構建工具** | Vite 5.x | 快速、現代、支援 HMR |
| **WebSocket** | ws 8.x | 輕量、穩定、易於使用 |
| **Notion API** | @notionhq/client | 官方 SDK、完整支援 |
| **測試框架** | Vitest | 快速、相容 Vite、易於使用 |

  ┌─────────────┐     ┌─────────────┐

  └─────────────┘     └─────────────┘

### 關鍵設計決策

| 決策 | 選項 | 選擇 | 原因 |
|------|------|------|------|
| **桌面框架** | Electron vs Tauri vs Qt | Electron | 開發快速、React 生態 |
| **狀態管理** | Redux vs Zustand vs Custom | Custom Hooks | 簡單場景不需要複雜狀態管理 |
| **樣式方案** | Tailwind vs CSS Modules vs Styled | CSS | 快速原型、易於調整 |
| **重連策略** | 固定間隔 vs 指數退避 | 指數退避 | 避免雪崩效應 |
| **Token 儲存** | 明文 vs 加密 | AES-256 加密 | 安全性考量 |

## 已知限制

### 技術限制

1. **WebSocket 連接數**: 受限於 ComfyUI 伺服器，建議最多 20 台
2. **Notion API 速率**: 3 requests/second，批次處理緩解
3. **平台支援**: 目前僅支援 Windows (Electron 可跨平台但未測試)

### 功能限制

1. **無遠端控制**: 只監控不控制
2. **無資源監控**: 不監控 GPU/記憶體使用率
3. **無複雜報表**: 僅基本歷史記錄

### 規模限制

1. **機器數量**: 建議 <20 台 (效能考量)
2. **歷史記錄**: 依賴 Notion 儲存空間
3. **用戶數量**: 單用戶設計
